1. 常量折叠

  当前 deob 保留了混淆器的算术混淆：

  _0x4d3a42 = _0x4d3a42 - (-0x2241 + 0x774 + 0x2 * 0xdfa);  // 20+ tokens, 需要心算

  一个简单的常量折叠 pass 就能变成：

  _0x4d3a42 = _0x4d3a42 - 295;  // 5 tokens, 直接可读

  这种"用复杂算式表达简单常数"的反混淆手段在文件中出现了数百次，每一次都消耗 LLM 额外的推理步骤去计算 -0x2241 + 0x774 + 0x2 * 0xdfa。这纯粹是 token 浪费——编译器优化里的入门技术，但对 LLM
  场景意外地关键。

  更激进的：如果能解析字符串（方案 D），连 _0x4d3a42 都可以被替换为它实际索引的字符串值，函数调用变成 a0_0x5465("getAttribute", ...) 甚至直接内联为 "getAttribute"。

  2. 标识符角色标注

  不追求重命名（太重），只做角色分类。每个 _0x 变量在函数中扮演什么角色：

  // 当前
  function _sub_ln180436_08_else(_0x54605c, _0x13f90f, a0_0x3fbebc, _0x5319d4, ...) {
    if (_0x54605c[_0x13f90f(...)](undefined, this[_0x5319d4(...)]))

  // 清洗后
  function _sub_ln180436_08_else(
    _0x54605c,     // [util:comparator]
    _0x13f90f,     // [decoder:string-resolver]
    a0_0x3fbebc,   // [config:table]
    _0x5319d4,     // [decoder:string-resolver]
    ...
  ) {
    if (_0x54605c[_0x13f90f(...)](undefined, this[_0x5319d4(...)]))
    //     ^-- util fn     ^-- decode str      ^-- decode str

  做法很简单：追踪每个参数在函数体中的使用方式：
  - 被当作函数调用 → [decoder] 或 [callback]
  - 被当作对象取属性 → [util] 或 [target]
  - 被当作参数传给 decoder → [string-key]
  - 未被使用 → 直接删除

  不需要知道具体叫什么，LLM 只需要知道"这个是解码器，那个是目标对象"就能大幅加速理解。

  ---
  3. 语义分块

  当前 deob 按语法边界提取函数（if 分支一个、else 分支一个）。更好的做法是按数据流边界聚类：

  一组函数都读写同一个 this.settings 对象 → 打包为 "Settings 模块"
  一组函数围绕 TextDecoder + RegExp → 打包为 "ContentFilter 模块"
  一组函数只做 a0_0x5465 调用传递 → 打包为 "StringDecode 基础设施"

  对 LLM 来说，一次分析一个完整的"功能模块"（5-15 个紧密相关的函数）比分别分析 196
  个孤立函数效率高得多。数据流分析可以自动发现这些模块边界——哪些函数共享同一组读写变量，哪些函数永远一起出现在调用链中。

  ---
  4. API 表面积预提取

  在 LLM 读函数体之前，先告诉它这个函数的"输入输出契约"：

  _sub_ln183442_08_else:
    reads: [this.configItems, _0x35c9ef.localize, _0x126bcf.DEFAULT_ICON]
    writes: []
    calls: [_sub_return_fn24, _sub_return_fn22]
    sideEffects: [none]
    summary: |
      Maps config items to {label, type, checked, enabled, icon, click} structures.
      Skips if configItems is empty. Uses localize() for type labels.
      Falls back to DEFAULT_ICON when item has no icon.

  LLM 可以先扫一眼 API 表面，决定是否值得读函数体。对于那 127 个穿透函数（pass-through），一行 reads: [], writes: [], sideEffects: [none] 就够了，不需要读代码。

  ---
  5. Token 预算分析 + 阅读顺序优化

  不是把反混淆结果 dump 给 LLM，而是做一个信息密度排序：

  Function                        Tokens  InfoDensity  Priority
  _sub_program_init_vars_ln1149   45000   ★★☆☆☆        P3 (先跳过，太费token)
  _sub_ln183442_08_else           800     ★★★★★        P1 (高密度，先看)
  _sub_constructor_08_if          200     ☆☆☆☆☆        P5 (穿透函数，跳过)
  a0_0x5465                       50      ★★★★★        P1 (核心调度器，必须看)

  生成一份"推荐阅读清单"，让 LLM 按优先级逐步加载。这比一股脑把所有文件丢进上下文窗口高效得多。可以设定 token 预算（比如 20K），自动选择在预算内信息密度最高的一批函数。

  ---
  7. 依赖拓扑排序

  当前 deob 输出按原始代码行号排列。LLM 读到第 100 行时遇到的函数，其依赖可能在第 5000 行才定义——上下文窗口里根本看不到。

  按调用依赖重新排序：被调用的函数先出现，调用者后出现。LLM 顺序阅读时自然构建心智模型，不需要来回跳转。

  L1:   a0_0x1cb6           (叶子，不调任何人)
  L10:  a0_0x5465           (调 a0_0x1cb6)
  L50:  _sub_return_fn1     (调 a0_0x5465["qOvhJb"])
  L200: _sub_ln113386_07_if (调 _sub_return_fn3, _sub_return_fn4)
  L300: _sub_return_fn3     (调 _sub_return_fn22, _sub_return_fn23)
  ...

  这利用了一个 LLM 的基本特性——前文影响后文。LLM 读到一个函数时，它的所有依赖已经在上面解释过了。

  ---
  8. 混淆模式知识库

  建立一个已知混淆模式的标注库，deob 自动匹配并标注：

  patterns:
    - name: "javascript-obfuscator string decoder"
      signature: "base64 table + RC4 KSA/PRGA + self-modifying accessor"
      annotation: "[STRING-DECODER: standard, skip]"

    - name: "anti-debug timing check"
      signature: "Date.now() comparison in loop, conditional debugger statement"
      annotation: "[ANTI-DEBUG: can ignore for logic analysis]"

    - name: "control-flow dispatcher"
      signature: "switch(state) with numeric states, each case assigns next state"
      annotation: "[CFG-DISPATCHER: flattened loop, see extracted cases for logic]"

    - name: "opaque predicate"
      signature: "complex arithmetic that always evaluates to true/false"
      annotation: "[DEAD-BRANCH: predicate always resolves to {value}]"

  LLM 看到 [STRING-DECODER: standard, skip] 就直接跳过整段代码，不需要自己推理"这段是不是解码器？我该不该读？"
